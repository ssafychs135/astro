RAG(Retrieval-Augmented Generation) 시스템에서 검색의 품질은 모델의 생성 능력만큼이나 중요합니다. 방대한 문서를 LLM이 이해할 수 있는 작은 단위로 나누는 **'청킹(Chunking)'**은 단순히 텍스트를 자르는 것이 아니라, 데이터의 **문맥(Context)**을 보존하는 고도의 전략적 작업입니다.

문서의 형태와 복잡도에 따른 4가지 핵심 청킹 전략과 실전 예시를 살펴봅니다.

## 1. 고정 크기 청킹 (Fixed-size Chunking)

가장 기본적인 방법으로, 글자 수나 토큰 수를 기준으로 일정한 크기로 문서를 나눕니다. 

> - **장점**: 구현이 매우 간단하고 계산 비용이 낮음.
> - **단점**: 문장이나 단어 중간이 잘려 문맥이 파괴될 위험이 큼.
> - **최적화**: '오버랩(Overlap)'을 설정하여 잘린 문맥을 보완합니다.

### 📝 예시
- **설정**: 크기 100자, 오버랩 20자
- **입력**: "AI 에이전트는 자율적으로 목표를 설정하고 도구를 사용하여 작업을 수행합니다. 이는 단순한 챗봇과는 차별화되는 특징입니다."
- **결과**:
  - 청크 1: "...자율적으로 목표를 설정하고 도구를 사용하여 작업을 수행합니다. 이는 단순한"
  - 청크 2: "(오버랩) 작업을 수행합니다. 이는 단순한 챗봇과는 차별화되는 특징입니다."

## 2. 슬라이딩 윈도우 청킹 (Sliding Window Chunking)

고정 크기 청킹의 '문맥 단절' 문제를 해결하기 위해, 창(Window)을 일정 간격(Stride)만큼 겹쳐가며 이동시키는 방식입니다.

> - **장점**: 인접한 청크 간의 문맥적 연결성(Contextual Continuity)을 강력하게 보장합니다. 검색 결과가 잘린 문장의 앞부분을 포함할 확률이 높아집니다.
> - **단점**: 동일한 텍스트가 여러 청크에 중복 저장되므로 저장 공간과 임베딩 계산 비용이 증가합니다.

### 📝 예시
- **설정**: 윈도우 크기 60자, 스트라이드(이동 거리) 30자
- **입력**: "RAG 시스템은 외부 데이터를 참조하여 답변을 생성합니다. 이를 위해 문서를 적절히 자르는 과정이 필수적입니다."
- **결과**:
  - 청크 1: "RAG 시스템은 외부 데이터를 참조하여 답변을 생성합니다. 이를 위해"
  - 청크 2: "(중복) 답변을 생성합니다. 이를 위해 문서를 적절히 자르는 과정이 필수적입니다."
  - *청크 1의 뒷부분과 청크 2의 앞부분이 겹쳐지며 문맥이 이어짐.*

## 3. 재귀적 문자 청킹 (Recursive Character Text Chunking)

문장 부호나 줄바꿈을 기준으로 우선순위를 두어 재귀적으로 분할합니다. 가장 널리 사용되는 범용적인 전략입니다.

> - **작동 방식**: `\n\n`(문단) -> `\n`(문장) -> ` `(단어) 순서로 구분자를 찾아가며 목표 크기에 맞춥니다.
> - **장점**: 문단의 구조와 문장의 무결성을 최대한 유지합니다.

### 📝 예시
- **입력**:
  ```text
  [문단 1: RAG의 정의]
  RAG는 검색 증강 생성의 약자입니다. 외부에 저장된 데이터를 참조합니다.

  [문단 2: 청킹의 중요성]
  데이터를 잘 나누어야 검색 정확도가 올라갑니다.
  ```
- **결과**: 문단 구분자(`\n\n`)를 최우선으로 고려하여, 문단 1과 문단 2를 독립적인 청크로 유지하되 크기가 넘치면 문장 단위로 분할합니다.

## 3. 문서 구조 기반 청킹 (Markdown/HTML-aware Chunking)

마크다운 헤더(`#`, `##`)나 HTML 태그를 인식하여 문서의 논리적 계층 구조를 보존합니다.

> - **장점**: 제목과 본문의 관계가 명확히 유지되어 검색 시 높은 관련성을 보장합니다.
> - **적용**: 기술 문서, API 가이드라인, 위키피디아 데이터.

### 📝 예시
- **입력**:
  ```markdown
  # 서비스 이용 약관
  ## 제 1조 (목적)
  본 약관은 서비스 제공을 목적으로 합니다.
  ```
- **결과**: 
  - `Metadata: {Header 1: "서비스 이용 약관", Header 2: "제 1조 (목적)"}`
  - `Content: "본 약관은 서비스 제공을 목적으로 합니다."`
  - *검색 시 제목 정보를 함께 참조하여 문맥 파악이 용이함.*

## 4. 의미론적 청킹 (Semantic Chunking)

텍스트의 의미적 유사도를 분석하여, 주제가 바뀌는 지점을 찾아 청킹합니다. 

> - **작동 방식**: 인접한 문장들 간의 임베딩 유사도를 측정하고, 유사도가 급격히 떨어지는 지점(Break point)을 경계로 나눕니다.
> - **장점**: 주제적 통일성이 완벽한 청크를 생성합니다.

### 📝 예시
- **입력**: "오늘 점심은 제육볶음을 먹었습니다. 정말 맛있더군요. 한편, 내일 있을 엔비디아의 실적 발표는 시장의 큰 관심을 끌고 있습니다."
- **결과**: 
  - 청크 1 (음식/일상): "오늘 점심은 제육볶음을 먹었습니다. 정말 맛있더군요."
  - 청크 2 (경제/기술): "한편, 내일 있을 엔비디아의 실적 발표는 시장의 큰 관심을 끌고 있습니다."
  - *두 문장 사이의 의미적 거리가 멀기 때문에 크기가 작더라도 분리함.*

## 🌟 결론: 정답은 '데이터'에 있습니다

최적의 청킹 전략은 정해져 있지 않습니다. 
- **뉴스/기사**: 재귀적 청킹
- **법률/약관**: 구조 기반 청킹
- **대화 데이터**: 고정 크기 + 큰 오버랩
- **전문 지식 베이스**: 의미론적 청킹

우선 작게 시작하여 검색의 **Recall(재현율)**을 테스트하며 우리 데이터에 맞는 최적의 구분자와 크기를 찾아가는 과정이 반드시 필요합니다.
