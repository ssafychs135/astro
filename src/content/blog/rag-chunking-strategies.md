RAG(Retrieval-Augmented Generation) 시스템에서 검색의 품질은 모델의 생성 능력만큼이나 중요합니다. 방대한 문서를 LLM이 이해할 수 있는 작은 단위로 나누는 **'청킹(Chunking)'**은 단순히 텍스트를 자르는 것이 아니라, 데이터의 **문맥(Context)**을 보존하는 고도의 전략적 작업입니다.

문서의 형태와 복잡도에 따른 4가지 핵심 청킹 전략과 실전 예시를 살펴봅니다.

## 1. 고정 크기 및 슬라이딩 윈도우 (Fixed-size with Sliding Window)

> - **작동 방식**: 윈도우 크기(Window Size)만큼 데이터를 담고, 다음 청크로 넘어갈 때 일정 간격(Stride)만큼만 이동하여 이전 청크의 끝부분을 포함시킵니다.
> - **장점**: 구현이 매우 간단하며, 오버랩을 통해 절단면에서 발생하는 정보 손실을 효과적으로 방지합니다.

### 📝 상세 예시
- **설정**: Window 100자 / Overlap 30자
- **입력**: "RAG 시스템은 외부 지식 베이스를 참조하여 답변의 신뢰성을 높입니다. 이를 구현하기 위해서는 문서를 적절한 크기로 자르는 청킹 과정이 필수적입니다. 청킹이 잘못되면 문맥이 파괴되어 엉뚱한 답변이 생성될 수 있습니다."
- **결과**:
  - **Chunk 1**: "RAG 시스템은 외부 지식 베이스를 참조하여 답변의 신뢰성을 높입니다. 이를 구현하기 위해서는 문서를 적절한 크기로 자르는 청킹 과정이 필수" (98자)
  - **Chunk 2 (Overlap 포함)**: "적절한 크기로 자르는 청킹 과정이 필수적입니다. 청킹이 잘못되면 문맥이 파괴되어 엉뚱한 답변이 생성될 수 있습니다." (전 청크의 마지막 30자 가량을 포함하여 문맥 유지)

## 2. 재귀적 문자 청킹 (Recursive Character Text Chunking)

> - **작동 방식**: `\n\n`(문단) -> `\n`(문장) -> ` `(단어) 순서로 구분자를 찾아가며 목표 크기에 맞춥니다.
> - **장점**: 문단의 구조와 문장의 무결성을 최대한 유지합니다.

### 📝 상세 예시
- **설정**: Target 150자
- **입력**: 
  ```text
  [Section 1]
  최신 AI 모델은 방대한 파라미터를 가집니다. 하지만 학습되지 않은 정보에는 취약합니다.

  [Section 2]
  이 문제를 해결하는 기술이 RAG입니다. RAG는 실시간으로 문서를 검색합니다.
  ```
- **결과**: 
  - 알고리즘이 먼저 `\n\n`을 감지하여 [Section 1]과 [Section 2]를 나눕니다. 
  - 각 섹션이 150자보다 작으므로, 문장 중간을 자르지 않고 두 개의 깔끔한 청크로 분리합니다. 
  - 만약 한 섹션이 200자라면, 그다음 구분자인 `.`이나 `\n`을 찾아 문장 단위로 다시 시도합니다.

## 3. 문서 구조 기반 청킹 (Markdown/HTML-aware Chunking)

> - **장점**: 제목과 본문의 관계가 명확히 유지되어 검색 시 높은 관련성을 보장합니다.

### 📝 상세 예시
- **입력**:
  ```markdown
  ## 요금제 안내
  - 베이직: 월 10,000원. 기본 기능 제공.
  - 프로: 월 30,000원. 모든 기능 제공.
  ```
- **결과**: 
  - **Chunk A**: `{Title: "요금제 안내", Content: "베이직: 월 10,000원. 기본 기능 제공."}`
  - **Chunk B**: `{Title: "요금제 안내", Content: "프로: 월 30,000원. 모든 기능 제공."}`
  - *단순 텍스트 자르기와 달리, 모든 청크가 "요금제 안내"라는 상위 맥락(Metadata)을 자동으로 상속받아 "베이직 가격이 뭐야?"라는 질문에 정확히 대응합니다.*

## 4. 의미론적 청킹 (Semantic Chunking)

> - **작동 방식**: 인접한 문장들 간의 임베딩 유사도를 측정하고, 유사도가 급격히 떨어지는 지점(Break point)을 경계로 나눕니다.

### 📝 상세 예시
- **입력**: "(문장1) 인공지능은 데이터 학습을 통해 성능을 개선합니다. (문장2) 딥러닝은 그 중 가장 핵심적인 기술입니다. (문장3) 내일은 전국적으로 비가 올 예정입니다. (문장4) 우산을 챙기시는 게 좋겠네요."
- **결과 분석**:
  - **Step 1**: 문장1-2 사이의 유사도 측정 -> 0.85 (높음)
  - **Step 2**: 문장2-3 사이의 유사도 측정 -> 0.12 (낮음, 주제 급변)
  - **Final**: [문장1, 2]를 'AI 기술' 청크로, [문장3, 4]를 '날씨' 청크로 분리. 글자 수가 적더라도 의미가 다르면 확실히 나눕니다.

## 🚀 Advanced: 성능을 극대화하는 고급 청킹 기술

단순히 자르는 단계를 넘어, 최신 RAG 아키텍처에서 사용하는 고도의 전략들입니다.

### 1. 컨텍스트 증강 청킹 (Contextual Retrieval)
각 청크 앞에 문서 전체의 핵심 맥락을 요약하여 덧붙이는 방식입니다. (Anthropic의 제안 기법)

> - **해결 과제**: "그는 1990년에 태어났다"라는 청크에서 '그'가 누구인지 모르는 문제를 해결합니다.
> - **작동 방식**: `[전체 요약] + [개별 청크]` 형태로 저장하여 검색 시 의미적 관련성을 비약적으로 높입니다.

### 2. 계층적 청킹 & 부모 문서 참조 (Parent Document Retrieval)
검색용 작은 청크(Child)와 요약용 큰 청크(Parent)를 분리하여 관리합니다.

> - **전략**: 
  - 작은 단위로 조밀하게 검색하여 관련 문서를 찾습니다.
  - 실제 LLM에게 전달할 때는 해당 청크가 속한 더 큰 '부모 문맥'을 통째로 전달합니다.
> - **장점**: 정교한 검색과 풍부한 답변 생성을 동시에 달성합니다.

### 3. 에이전트 기반 청킹 (Agentic Chunking)
LLM 에이전트가 문서를 읽으며 스스로 "여기서부터는 새로운 주제군!"이라고 판단하게 하는 방식입니다.

> - **특징**: 고정된 알고리즘이 아닌, AI의 추론 능력을 사용하여 비정형 데이터(보고서, 논문 등)에서 최적의 분할 지점을 동적으로 찾아냅니다.

## 💡 실전 시나리오: 청킹과 리트리버의 최적 조합

이론을 넘어, 실제 서비스에서는 데이터의 성격에 따라 청킹과 리트리버를 어떻게 조합해야 할까요? 대표적인 3가지 사례를 통해 살펴보겠습니다.

### 시나리오 A: 고객 센터 상담 챗봇 (비정형 대화 데이터)
- **데이터 특성**: 짧은 문장 위주, "어제 산 물건"과 같은 일상적 표현 많음.
- **최적 조합**: `슬라이딩 윈도우 청킹` + `밀집 리트리버 (Dense)`
- **🎯 실전 사례**:
  - **질문**: "배송이 너무 늦는데 언제 오나요?"
  - **작동**: 슬라이딩 윈도우로 잘린 상담 가이드 청크들이 '배송 지연', '예상 도착일' 등의 의미를 담은 벡터로 검색되어, 질문에 "배송"이라는 단어가 없어도 가장 유사한 가이드를 찾아냅니다.

### 시나리오 B: 사내 법률 및 약관 검색 시스템 (구조화된 문서)
- **데이터 특성**: 제1조, 제2항 등 엄격한 계층 구조, 전문 용어 집약적.
- **최적 조합**: `문서 구조 기반 청킹` + `하이브리드 리트리버 (Hybrid)`
- **🎯 실전 사례**:
  - **질문**: "개인정보 처리방침 제5조 3항의 예외 조항 알려줘"
  - **작동**: 마크다운 헤더로 잘린 청크가 '제5조 3항'이라는 키워드를 희소 리트리버(BM25)로 정확히 짚어내고, 동시에 "예외 조항"의 의미를 밀집 리트리버가 보완하여 가장 정확한 법적 근거를 추출합니다.

### 시나리오 C: 대규모 기술 논문 분석 에이전트 (전문 지식)
- **데이터 특성**: 한 문서 내에 여러 실험 결과와 수식이 섞여 있음. 문맥 파악이 매우 어려움.
- **최적 조합**: `의미론적 청킹` + `리랭킹 리트리버 (Re-ranking)`
- **🎯 실전 사례**:
  - **질문**: "이 논문에서 제안한 아키텍처의 한계점이 뭐야?"
  - **작동**: 주제가 바뀌는 지점을 정확히 나눈 의미론적 청크들을 먼저 검색한 뒤, 리랭커(Cross-Encoder)가 검색된 50개의 후보 중 '한계점'과 '비판적 시각'이 가장 뚜렷하게 나타난 청크를 최상단으로 재정렬하여 LLM에게 전달합니다.

## 🌟 결론: 정답은 '데이터'에 있습니다

최적의 청킹 전략은 정해져 있지 않습니다. 
- **뉴스/기사**: 재귀적 청킹
- **법률/약관**: 구조 기반 청킹
- **대화 데이터**: 고정 크기 + 큰 오버랩
- **전문 지식 베이스**: 의미론적 청킹

우선 작게 시작하여 검색의 **Recall(재현율)**을 테스트하며 우리 데이터에 맞는 최적의 구분자와 크기를 찾아가는 과정이 반드시 필요합니다.
