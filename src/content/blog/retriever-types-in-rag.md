RAG(Retrieval-Augmented Generation) 시스템에서 청킹이 데이터를 '준비'하는 과정이라면, **리트리버(Retriever)**는 준비된 데이터에서 최적의 정보를 '찾아내는' 핵심 엔진입니다. 단순히 검색하는 것을 넘어, 사용자의 질문 의도를 파악하고 가장 관련성 높은 문맥을 LLM에게 전달하는 4가지 주요 리트리버 전략을 소개합니다.

## 1. 희소 리트리버 (Sparse Retriever)

> - **작동 원리**: 질문에 포함된 특정 단어가 문서에 얼마나 자주 등장하는지를 기준으로 점수를 매깁니다.
> - **강점**: 고유 명사, 전문 용어, 정확한 제품명이나 수치 검색에 오차가 거의 없습니다.

### 📝 상세 예시
- **질문**: "2024년형 갤럭시 S24 울트라의 배터리 용량 mAh로 알려줘"
- **결과**: "갤럭시 S24 울트라", "2024", "배터리", "mAh"라는 키워드가 모두 포함된 특정 페이지를 정확히 찾아냅니다. 
- **특징**: 키워드가 하나라도 틀리면 검색 결과에서 밀려날 수 있지만, 정확한 키워드 입력 시 가장 신뢰도 높은 결과를 줍니다.

## 2. 밀집 리트리버 (Dense Retriever)

> - **작동 원리**: 질문과 문서를 숫자로 이루어진 벡터로 바꾸고, 벡터 공간에서의 거리(유사도)를 계산합니다.
> - **강점**: 유의어, 동의어 검색이 가능하며 전반적인 맥락을 이해합니다. (Semantic Search)

### 📝 상세 예시
- **질문**: "외출할 때 비가 올 것 같은데 준비물이 뭐야?"
- **결과**: 문서 내에 "비", "준비물"이라는 단어가 없더라도, "강수 확률", "우산", "레인코트" 등의 의미적 연관성이 높은 청크를 벡터 유사도로 찾아냅니다.
- **특징**: 사용자가 전문 용어를 몰라도 일상어로 정답에 접근할 수 있게 해줍니다.

## 3. 하이브리드 리트리버 (Hybrid Retriever)

> - **전략**: 두 리트리버의 결과 점수를 적절한 가중치로 합산(Reciprocal Rank Fusion 등)하여 최종 순위를 결정합니다.

### 📝 상세 예시
- **질문**: "아이폰 전원이 안 켜질 때 해결법"
- **결과 추출 과정**:
  - **Sparse**: "아이폰", "전원" 키워드로 관련 매뉴얼 10개 추출.
  - **Dense**: "안 켜질 때", "해결법"의 의미를 분석하여 "강제 재부팅", "충전 확인" 등의 내용이 담긴 청크 추출.
  - **Final**: 두 검색 결과에서 공통적으로 상위에 오른 '아이폰 강제 재시작 가이드'가 최종 1순위로 선정됨.

## 4. 리랭킹 리트리버 (Re-ranking Retriever)

> - **작동 방식**: 먼저 빠르고 가벼운 리트리버로 후보군을 뽑고, 강력한 모델로 순위를 다시 매깁니다.

### 📝 상세 예시
- **질문**: "파이썬에서 리스트를 정렬하는 가장 효율적인 방법은?"
- **1차 검색**: `sort()`, `sorted()`, `heapq`, `bubble sort` 등 정렬 관련 청크 50개 추출 (속도 중시).
- **2차 리랭킹**: Cross-Encoder 모델이 50개를 전수 조사하여, 질문의 핵심인 "가장 효율적인(시간 복잡도)" 관점에서 `sort()`(Timsort)의 성능을 다룬 청크를 최상단으로 올리고, 단순 구현인 `bubble sort`는 아래로 내림.

## 🚀 Advanced: 검색 품질을 혁신하는 고급 리트리버 기술

기본적인 검색 방식을 넘어, LLM의 추론 능력을 검색 과정 자체에 개입시키는 고도의 전략들입니다.

### 1. 멀티 쿼리 리트리버 (Multi-query Retrieval)
사용자의 모호한 질문을 LLM이 여러 개의 다양한 관점으로 재해석하여 검색하는 방식입니다.

> - **전략**: "하나의 질문으로 한 번 찾기" 대신, "5개의 유사한 질문으로 5번 찾기"를 수행합니다.
> - **효과**: 질문의 미묘한 단어 선택 차이로 인해 검색 결과가 누락되는 현상을 방지합니다.

### 2. HyDE (Hypothetical Document Embeddings)
사용자의 질문에 대해 LLM이 먼저 '가상의 답변'을 생성하고, 그 가짜 답변과 유사한 진짜 문서를 찾는 방식입니다.

> - **전략**: `질문 -> 가상 답변(LLM) -> 벡터 검색 -> 진짜 문서`
> - **효과**: 질문과 답변 사이의 의미적 거리(Gap)를 줄여 검색 정확도를 비약적으로 높입니다.

### 3. 컨텍스트 압축 (Contextual Compression)
검색된 문서들 중에서 사용자 질문과 정말로 관련 있는 '핵심 문장'만 추출하여 LLM에게 전달하는 기술입니다.

> - **전략**: 문서 통째로 넘기는 대신, 리트리버가 질문과의 관련성을 기준으로 본문을 필터링하거나 요약합니다.
> - **효과**: LLM에게 전달되는 노이즈를 줄이고 토큰 비용을 절감합니다.

### 4. 셀프 쿼리 리트리버 (Self-query Retrieval)
자연어 질문에서 메타데이터 필터(날짜, 카테고리, 점수 등)를 자동으로 추출하여 검색 쿼리에 적용하는 방식입니다.

> - **📝 예시**: "2023년 이후에 발행된 AI 관련 뉴스 알려줘"라고 물으면, `text: "AI", filter: {year: ">2023"}`로 자동 변환하여 정교하게 검색합니다.

## 🌟 결론: 어떤 리트리버를 선택해야 할까요?

리트리버의 선택 역시 보유한 데이터의 성격에 따라 달라집니다.

- **정형화된 매뉴얼/사양서**: 희소 리트리버 중심
- **자유로운 대화/지식 베이스**: 밀집 리트리버 중심
- **기업용 복합 지식 관리**: 하이브리드 + 리랭킹 (권장)

결국 RAG의 성능은 **"얼마나 잘 잘랐는가(Chunking)"**와 **"얼마나 잘 찾아내는가(Retriever)"**의 정교한 조합에서 결정됩니다.
